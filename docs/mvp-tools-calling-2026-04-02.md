# MVP Tools Calling Snapshot — 2026-04-02

## Статус

Это архивный снимок ветки `MVP_tools_calling`, зафиксированный как состояние проекта на `2026-04-02`.

Эта ветка сохраняется отдельно как исторический MVP-срез, чтобы не потерять уже сделанную работу и иметь точку сравнения перед правильной переработкой.

## Важная пометка

Текущая реализация agent-layer в этой ветке считается промежуточной и архитектурно неправильной для целевого решения.

Что именно не так:

- orchestration для tools и agent loop написаны вручную с нуля;
- observability и tracing не выстроены на готовом специализированном слое;
- часть поведения LLM/provider-слоя пришлось стабилизировать локальными обходами;
- из-за этого поддержка, отладка и дальнейшее развитие будут дороже, чем должны быть.

## Что стоило использовать

Для следующей правильной реализации нужно опираться на готовые инструменты, а не продолжать развивать самописный orchestration-слой:

- `LangChain` или близкий готовый agent/tool framework для tool calling, orchestration и standard abstractions;
- `Langfuse` для traces, наблюдаемости, prompt/version tracking и анализа поведения агента.

## Почему так получилось

Самописный MVP был собран по незнанию и для быстрого движения вперёд: на этом этапе мы реализовали agent loop, tools и UI вручную, вместо того чтобы сразу поставить правильный foundation на готовых фреймворках.

Это нормально как временный исследовательский шаг, но не как целевая архитектура.

## Как относиться к этой ветке дальше

- не считать её финальной архитектурой;
- использовать как reference snapshot;
- сравнивать с будущей правильной веткой, где agent-layer будет пересобран на готовых инструментах.
